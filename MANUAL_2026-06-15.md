# PoolTerminal — Operator Manual

**Manual version 1.2 — updated 15 June 2026**

A practical guide to running and understanding PoolTerminal. For the project
overview see `README.md`; for architecture and design rationale see `DESIGN.md`.

---

## 1. What PoolTerminal is

A read-only, Bloomberg-style desktop terminal for Cardano SPOs. It connects to
your node (over SSH, or directly when run on the node) and to read-only public
data sources, pulls together operational data, visualises it, and caches history
locally so you can see trends over time.

It never signs transactions, never touches keys, and never issues commands that
change node state. Secrets you enter to connect (SSH password, OTP, key
passphrase) are held only in memory for the session and are never written to disk.

## 2. Connecting

The connect dialog offers a **Connection type**:

### 2.1 Remote node (SSH)

For a node on another machine. Fill in host, port, username, and the path to your
Guild Operators **env file** on the node (e.g. `/opt/cardano/cnode_bp/scripts/env`).
Choose an **authentication method**:

- **Password / 2FA** — plain password, or keyboard-interactive for nodes with
  2FA (choose the order: code-then-password etc.).
- **SSH key file** — pick an auto-detected key from `~/.ssh`, or enter a custom
  path (e.g. a cloud `.pem`). Encrypted keys are unlocked with a passphrase you
  enter (never stored).
- **SSH agent** — uses your running `ssh-agent` (the same keys `ssh` uses in a
  terminal). Recommended for cloud hosts. Make sure your key is loaded
  (`ssh-add`).

### 2.2 This machine (local)

For running PoolTerminal **on the node itself**. No host, port, or auth needed —
it runs commands locally. Just give the env file path.

Your remembered **remote** connection details (SSH host, port, username) are
preserved when you use local mode, so switching back to SSH later still points at
the right node. External links (Cardanoscan transactions, the project site) open
in your system browser.

### 2.3 Block producer vs relay

PoolTerminal detects the node's role. On a **relay**, block-producer-only data
(KES, ideal, leader, upcoming blocks) can't exist — those hero cards are labelled
"(relay)" and the upcoming-blocks panel reads "Operating in relay mode only", so
it's obvious why they're blank. Relays connect cleanly without waiting on data
they can't provide.

## 3. The views

### 3.1 NOW — live current-epoch dashboard

- **Chain pulse** — live tip / slot heartbeat.
- **Hero row** — epoch, KES expiry, ideal & leader, adopt/confirmed/lost,
  delegators, sync.
- **Upcoming blocks** — your assigned leader slots, current epoch *and* next epoch
  (once the ~36 h leadership-schedule window opens), each with a live countdown
  and an epoch tag. Sourced from `cardano-cli query leadership-schedule` (the
  authority — not cncli, whose leaderlog can carry restore artifacts). Cached per
  epoch in the local DB, so it computes at most once per epoch and repaints
  instantly when you return to the tab.
- **Mempool, peers, relay map** — live.

The loading overlay waits until the universal data (chain pulse, peers) and — on a
BP — KES and ideal are populated, then transitions to a fully-filled screen.

### 3.2 HISTORY — per-epoch table & charts

Full history back to your pool's first epoch: blocks, ideal, luck, delegators,
active stake, and a six-way reward split, plus blocks-per-epoch and luck charts.
The reward columns are colour-coded for quick reading:

- **Deleg Rwd** (blue) — rewards paid to delegators (member rewards).
- **Pledge Rwd · Min Fee Rwd · Margin Rwd** (green) — the three components of the
  operator's own take: reward on pledged stake, the fixed min-pool-cost (min fee),
  and the margin earned on the rest.
- **SPO Earnings** (bold green) — the sum of Pledge + Min Fee + Margin rewards,
  i.e. exactly what the operator earned from the epoch (equals the leader reward).
- **Total Payout** (bold blue) — delegator + operator combined, reconciling with
  PoolTool.

The panel header shows the active **data source** and a temporary **Switch
source** button (db-sync ↔ Koios) for testing — see §4. The choice persists and
takes effect on reload.

### 3.3 DELEGATORS — unified delegator + loyalty table

One data-rich, sortable table of every current delegator, merging live-stake data
with a computed **loyalty** ranking. Each row shows: rank, stake address (your own
flagged "you"), a loyalty bar (blue tenure segment + gold stake-weight segment),
the loyalty score, tenure, stake-weight, **penalties**, live stake, and % of pool.

- **Sort** by Loyalty (default) or by Stake, with a dust filter (Hide < 5 ₳) and
  true Prev/Next pagination (100 rows per page).
- **Loyalty score** = tenure-rank × stake-weight × defection penalty × reduction
  penalty (see §4.9). Computed whole-pool from db-sync, then **cached per epoch**
  in the local DB — the heavy compute runs at most once per epoch (instant on
  every subsequent open).
- **Deep-dive** — click any delegator for a modal: balance, rewards earned,
  withdrawn, since-epoch, DRep flag, origin pool, and a colourful **pool-movement
  journey** — each pool they've delegated to as a node showing entry (epoch +
  active stake) and exit (epoch + active stake), your pool gold-highlighted with
  "still here", the chain self-verifying (each pool's exit epoch = the next's
  entry). Sourced from Blockfrost `/accounts/{stake}/history` (real per-epoch
  active stake, matches db-sync `epoch_stake`).

The delegator list and loyalty data come from **db-sync** (one fast query, no
rate limits, no Blockfrost key needed for the list). The deep-dive currently needs
a Blockfrost key; a db-sync deep-dive path is planned.

### 3.4 NODE HEALTH

Host metrics (CPU, memory, disk) and node-process metrics (peers in/out, resource
use), with historical samples.

### 3.5 MAP

Full-size equirectangular world map (Natural Earth, cached offline after first
load): your node as a gold anchor, live peers geo-located with RTT-coloured
connection lines, and a side panel with latency bands (fast/medium/slow/unknown)
and a ranked geographic distribution of peers. Hover a peer for IP, city/country,
AS, and RTT. (The small NOW-tab relay widget is separate and unchanged.)

### 3.6 NOTIFICATIONS — live delegation feed

A real-time feed of stake movements in and out of your pool, detected on-chain
and shown newest-first with colour-coded rows. Event types:

- **New delegator** — a stake address delegating to you for the first time.
- **Transfer in** — a delegator moving to you from another pool (their previous
  pool's ticker is shown).
- **Returning delegator** — a stake address that was *ever* delegated to you
  before, coming back. Distinguished from a brand-new delegator.
- **Redelegated away** — a delegator moving from you to another pool (their new
  pool is shown).
- **Stake added / Stake reduced** — a meaningful change in an existing
  delegator's live stake.

Each row shows a from→to flow (your pool's pill is **green when stake is coming
in, red when going out**; other pools get a stable colour from their ticker), the
amount, epoch, slot, UTC date/time, and a **tx** link that opens the transaction
on Cardanoscan. Persistent corner toasts announce activity from any tab; the
Notifications tab carries an unread badge.

**Detection.** With **db-sync** connected, joins, returns and redelegations-away
are detected within minutes (every delegation certificate is read directly from
the chain). **Koios-only** operators get the same feed but at epoch-boundary
cadence, since Koios's delegator list only updates each epoch. Stake up/down is
computed by comparing live balances poll-to-poll. Every event is stored once and
de-duplicated at the database level, so the same on-chain action can never appear
twice (even across watermark resets).

Notification history lives in the local cache. To clear the feed and start fresh,
stop the app and run:
`sqlite3 ~/.local/share/com.gnp1.poolterminal/poolterminal.db "DELETE FROM notif_events;"`

## 4. Data sources

History is served from one of two interchangeable sources behind a single
selector. Both produce the same canonical per-epoch rows, so the views don't care
which is active.

### 4.1 db-sync

Reads a Cardano **db-sync** Postgres directly — local (Unix socket, peer auth) or
remote (TCP with credentials). Complete, gap-free, no rate limits. Best if you
already run db-sync. Schema-version aware (validated against a known schema;
flags drift).

### 4.2 Koios (no-infrastructure fallback)

The entire backfill is **two bulk API calls** — `pool_history` (all epochs) and
`epoch_info` (network stake + block count per epoch) — so it is rate-limit-safe by
design. No per-epoch loops (the thing that gets you banned).

### 4.3 Choosing / switching

The HISTORY header shows "Source: dbsync|koios" and a **Switch source** button.
Switching persists (survives restart) and reloads so the chosen source backfills
fresh. Note: switching doesn't wipe the previous source's cached rows — the new
source overwrites the epochs it covers as it loads, so give it a moment to settle.
A future setup wizard will replace this temporary toggle.

### 4.4 How Ideal is computed (and the active-stake-pct gap)

Ideal = σ × expected-blocks, where σ is the pool's share of **active (delegated)
stake** — not total supply.

- **Current epoch:** σ from `cardano-cli stake-snapshot` (`stakeGo` ÷ total
  `stakeGo`) × 21600 (epoch_length × active-slot-coefficient). cli, authoritative.
- **Historical epochs:** Koios `pool_history.active_stake_pct` would give σ
  directly, **but it is null for all but the most recent epochs** (audit: 394 of
  409 epochs null), so it can't be relied on. Instead:
  - pool active stake (lovelace) from `pool_history.active_stake` (present for all);
  - network active stake + real block count from `epoch_info`
    (`active_stake`, `blk_count`);
  - `σ = pool_active_stake / network_active_stake`; `ideal = σ × blk_count`.
  - Verified: epoch 300 → σ = 308,936,127,061 / 23,882,888,816,746,304 = 1.29e-5;
    ideal = σ × 21,225 = 0.27 (tiny early pool, 0.46 % saturation — correct).

For **db-sync**, ideal is filled for the whole history in two bulk queries
(net-stake `GROUP BY` epoch + net-blocks per epoch), including zero-block epochs
(their luck is a meaningful 0 %). It runs once after backfill (~16 s for ~400
epochs) and is then cached forever.

Past epochs are finalised, so each Ideal is computed once and cached forever.

**Avg luck** is `lifetime blocks ÷ lifetime ideal` (the honest ratio). A mean of
per-epoch luck ratios over-weights low-ideal epochs and reads misleadingly high.

### 4.5 SPO pledge & reward split

Koios's `pool_history` gives `member_rewards` (delegators, excluding pledge) and
`deleg_rewards` (all delegated-stake rewards, including the operator's pledge), so
the **pledge reward = deleg_rewards − member_rewards**. PoolTerminal sets
`leaderReward = pool_fees + pledge reward` so the HISTORY table's split (pledge =
leader − fee − margin) and **Total payout** reconcile with PoolTool. (db-sync's
`reward` table already carries the leader reward inclusive of pledge.)

### 4.6 Known Koios limitation

Koios's `pool_history` **occasionally returns null reward fields for a
block-producing epoch** (e.g. epoch 627: `block_cnt=1` but `pool_fees`,
`deleg_rewards`, `member_rewards` all null), while db-sync has the full data.
PoolTerminal shows `—` for those cells on Koios — truthful, since Koios didn't
return the values, rather than inventing a number. Operators wanting gap-free
history use db-sync; Koios is the no-infrastructure fallback with the occasional
reward-data hole. This is **not a PoolTerminal bug**.

### 4.7 Transport options (planned wizard choice)

Node-side telemetry (cardano-cli, Prometheus) is **always** run on the node. What
is selectable is where **public-API calls (Koios, geo)** run. The connect wizard
will offer three modes with plain-English explanations:

| Mode | How public-API calls run | Best for | Caveat |
|---|---|---|---|
| **Via the node** (default today) | `curl` over SSH on the node | Node has internet egress | Fails on an egress-locked BP |
| **Via this machine** | Host hits the API directly | Hardened BPs with no egress; frees the SSH channel | Needs the workstation to have internet |
| **Offline / seed** | No live external calls; history from a pre-built seed DB | Air-gapped setups | History only as fresh as the last seed |

Koios-on-node depends on the *node* having egress — exactly what a hardened BP is
configured not to have; Koios-from-host depends on the workstation having internet
— which it essentially always does. Neither is universally right, so the operator
chooses. The transport is isolated to one function (`runCmd` in `koios-query.js`),
so adding host and seed transports is additive, not a rewrite.

### 4.8 Read-model collector jobs

All run fire-and-forget from the ~1 Hz loop, each internally throttled; none gates
a snapshot. The node is read-only; only the local SQLite cache is written.

| Job | Cadence | Writes | Notes |
|---|---|---|---|
| backfill | once (version-gated) | all epoch rows | One bulk fetch. Bump the version const to force an in-place rebuild (INSERT-OR-REPLACE, no wipe). |
| ideal filler | small batches | `ideal` on null rows | Historical ideal (§4.4). Stops when none remain; cached forever. |
| recent refresh | ~5 min | newest ~6 epochs | Picks up epochs not yet published at backfill (e.g. the just-ended epoch). |
| semi-live | ~3 min | time-series samples | live stake, delegators, saturation, pledge. |
| block production | per epoch + periodic | current epoch row + cards | leadership-schedule once per epoch; produced count periodically; dirty-guarded writes. |

### 4.9 Loyalty scoring (DELEGATORS)

Each delegator's loyalty score (0–100) is:

```
loyalty = tenureScore × stakeWeight × defectionPenalty × reductionPenalty
```

- **tenureScore** — the delegator's current unbroken run length (consecutive
  epochs delegated to your pool, via a gaps-and-islands query) relative to the
  longest-serving delegator, ×100. Rewards sustained loyalty, not raw age.
- **stakeWeight** — `curStake / (curStake + 10000)` using **current** stake. Dust
  collapses toward 0; whales saturate toward 1. Uses current (not lifetime-average)
  stake so an ex-whale who has withdrawn is scored on what they hold now.
- **defectionPenalty** — `0.25` if the delegator ever moved ≥50% of their stake to
  **another pool** (a true defection, permanent), else `1`. A withdrawal to wallet
  (no redelegation) is **not** a defection.
- **reductionPenalty** — for big withdrawals to wallet (≥50% drop, no redelegation):
  `1 − min(1, reductionFactor)×0.5` (max 50% dock), where the factor decays with a
  ~36-epoch (~6 month) half-life, so an old withdrawal fades while a recent one
  still bites.

Computed whole-pool from db-sync `epoch_stake` in one query, then cached in the
`delegator_loyalty` table keyed by epoch — heavy compute runs at most once per
epoch. The ↻ button forces a recompute.

## 5. Caching & the local database

PoolTerminal keeps a local SQLite cache (`poolterminal.db`):

- **Epoch history** — one row per epoch; written once and reused. Survives
  restarts; switching source overwrites covered epochs.
- **Leader schedule** — per epoch (`leaderlog:<epoch>:<current|next>`), so the slow
  `cardano-cli` schedule query runs at most once per epoch.
- **Net stake** — cached per epoch (the heavy db-sync ideal denominator).
- **Delegator loyalty** — the whole-pool loyalty snapshot (`delegator_loyalty`
  table), keyed by epoch, so the heavy loyalty compute runs at most once per epoch.
- **Node-health samples** — time series.
- **Notification events** — the delegation feed, de-duplicated by a unique
  natural key so the same on-chain action is never stored twice.

The schema is versioned (currently v5); migrations are additive and never wipe
existing cached data.

The cache is on *your* machine only. Nothing is sent anywhere.

## 6. Troubleshooting

- **Scaffold screen ("Phase 0 — scaffold ready")** — a frontend module failed to
  load. Open dev tools (F12) → Console, read the red error; it names the file/line.
- **Upcoming blocks empty on a BP** — confirm you're connected to the **BP** (not a
  relay) and that you're within the leadership-schedule window. The current epoch
  may genuinely have no remaining slots.
- **Upcoming blocks shows on a relay** — shouldn't; if it does, the node role
  probe may be misreporting. It's gated to BP role.
- **Ideal / Luck blank in HISTORY** — on first load the db-sync ideal fill takes
  ~16 s, then caches. On Koios, some epochs legitimately lack reward data (§4.6).
- **Koios rate limit (HTTP 429)** — shouldn't happen (2 bulk calls), but if you see
  it, the window is shared/global on Koios's side and drains over time.
- **Connect dropdown text invisible (older WebKitGTK)** — fixed via CSS; if a build
  still shows it, it's a theme-rendering quirk of that WebKitGTK version.

## 8. Security recap

- Read-only; no state-changing commands.
- Keys never leave the node; PoolTerminal never reads or transmits them.
- Connection secrets (password / OTP / passphrase) held in memory only, never
  written to disk.
- Network egress: your node (SSH), your db-sync (if used), and read-only public
  APIs (Koios, ip-api for peer geo). No telemetry.
